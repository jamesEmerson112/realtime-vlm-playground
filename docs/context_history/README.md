- [vlm-prompt-v3] VLM prompt (`prompts/vlm_prompt.txt`) refactored away from pure-observer toward **structured per-frame signal**. New placeholders: `{completed_steps}`, `{current_step_block}`, `{next_step_block}` (replace old `{steps_text}` + `{current_step_line}`). New JSON fields the VLM now emits: `end_state_visible` ("yes" | "no" | "unclear"), `heard_in_audio_matches_current_step` (bool), `error_visible` (description or null). Mother still owns final event decisions but now consumes pre-digested signals instead of raw free-text descriptions. Rationale: gives the mother structured evidence to reason over (e.g., "at t=57s, end_state_visible=yes for step 2" is a much stronger signal than "hands closing panel door").

Rationale: lets the mother see cause-effect pairs inline (VLM at t=56s says "error_visible: wrong toolbox" / audio at t=57s says "no, not that one") instead of having to cross-reference two separate lists. - [rule-based-advance] New `Pipeline._check_and_advance_step(observation, audio_window)` method. Votes on two signals per observation: (Rule B) audio-phrase match — any step keyword appears in any transcript across the 3-chunk 10s audio window; (Rule C) end-state match — `end_state_visible == "yes"` OR any step keyword appears in the VLM's `objects` + joined `visual_cues` text. Two consecutive positive votes advance `current_step_index`, add the step to `completed_steps`, reset the vote counter, and log a `state_change` entry with `trigger: {audio_match, end_state_match}`. When the 2-of-3 voter fires, the step is stored as "pending" and the queue advances. The `step_completion` event is only emitted when the VLM confirms the NEXT step is happening (`current_step_happening=yes`). Prevents cascade desync entirely — step N can't be emitted prematurely because we need proof step N+1 started. Last step emits immediately (no next step to confirm). Timeout fallback at 15s.
